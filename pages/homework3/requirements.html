Homework 3

For this homework assignment, you must visualize data from the SF 311 Cases dataset using a geospatial data visualization technique in D3 version 5. The techniques allowed for this homework include: non-proportional symbol map, proportional symbol map, choropleth map, or hexbin map.

1. Dataset

You must visualize a filtered view of the SF 311 dataset available at:

https://data.sfgov.org/City-Infrastructure/311-Cases/vw6y-z8j6 (Links to an external site.)

Be sure to read the 311 case data FAQ (Links to an external site.) before visualizing the data .

The visualizations must use the geospatial data columns (latitude and longitude for symbol and hexbin maps, district or neighborhood for choropleths).

You may show the raw count of cases or attempt to calculate a case per sq ft (Links to an external site.) ratio instead. Either way, the data being visualized should be clearly indicated.

You are also encouraged to explore using the many categorical columns or the time series columns (such as calculating the time between opened and closed) available in the dataset.

1.1 Filtering

Filter the dataset to show cases that were OPENED in the month of February 2020 only. Then choose a CATEGORY that has at least 100 cases associated with it. For example, this includes "Street and Sidewalk Cleaning" (approximately 21k cases) and "Residential Building Request" (approximately 100 cases).

Optional: While some "General Request" subcategories have few cases (e.g. "General Request - ANIMAL CARE CONTROL" has approximately 60 cases), you can treat any of the categories that start with "General Request" as one larger category.

1.2 Base Map

Decide on a map projection to use that is appropriate for their data and visualization technique.

Show district or neighborhood boundaries (or a mix) no matter which projection or visualization technique you choose. The options include:

  • Current Supervisor Districts: https://data.sfgov.org/Geographic-Locations-and-Boundaries/Current-Supervisor-Districts/8nkz-x4ny (Links to an external site.) 

  • Current Police Districts: https://data.sfgov.org/Public-Safety/Current-Police-Districts/wkhw-cjsf (Links to an external site.) 

  • SF Find Neighborhoods: https://data.sfgov.org/Geographic-Locations-and-Boundaries/SF-Find-Neighborhoods/pty2-tcw4 (Links to an external site.)

If you want to show other geographic context (such as streets), you can find some additional options at:

https://data.sfgov.org/browse?category=Geographic+Locations+and+Boundaries (Links to an external site.)

All the boundary files may be exported in GeoJSON format.

1.3 Additional Resources

There is a basic non-proportional symbol map visualization of this dataset available at:

https://data.sfgov.org/d/ky99-f5iy (Links to an external site.)

It is very slow and should be viewed with a filter applied. There is also a dashboard overview, which may be helpful in determining the filter you want to apply:

https://data.sfgov.org/City-Infrastructure/311-Cases-Dashboard/tyzs-6f38 (Links to an external site.)

The dashboard also includes an example of a choropleth using this data.

2. Requirements

To earn credit, you must both satisfy one of the functionality levels below and provide the required writeups.

 Letter  Requirement
D A visualization prototype in any other tool or language than D3v5.
C A static visualization in D3v5 with a projected district or neighborhood base map.
B The C level requirements plus details-on-demand interactivity.
A The B level requirements plus either zoom, filter, or brush interactivity.

The C, B, and A levels do not require prototyping, but it is encouraged.

The homework website must have clear navigation to the required sections. Visualizations should be approximately 960 by 500 pixels wide, although some exceptions are reasonable. 

2.1 Functionality

The functionality by letter grade is as follows:

2.1.1 D Level Functionality

Visualize the dataset (see §1.1 for details) on a projected base map (see §1.2 for details) using one of the following visualization techniques:

  • Non-Proportional Symbol Map: This option should show the location of every individual request made. You should encode an additional piece of categorical information using color.

    For example, you can color by the "Source", "Request Type", "Status", or "Responsible Agency" columns. Also, consider how to use shape, stroke, and opacity of the symbols effectively.

  • Proportional Symbol Map: Nest and rollup the data by either district or neighborhood, and visualize a numerical value using the size of each symbol. Choose the appropriate underlying boundary map based on your choice.

    You will also need to calculate the "center" of that region to place each the symbol. You should be able to use the centroid method (Links to an external site.) in D3, but it might be easier to just find the bounding box (Links to an external site.) for a region and calculate the center pixel location.

  • Choropleth Map: Nest and rollup the data by either district or neighborhood, and visualize a numerical value using fill color for that region.

    Choose the appropriate underlying boundary map based on your choice. Consider what color scales make the most sense for your data, and whether to use a threshold scale or linear scale.

    Ideally, you should calculate a per sq ft or per capita ratio FIRST, but this is not required since population data for this year is difficult to find.

  • Hexbin Map (Challenging): Use the d3-hexbin library (Links to an external site.) to bin the incidents and display those bins. You can choose to color or size the bin to show the number of incidents that occurred in that bin.

Choose whichever technique you are more interested in–all are worth the same amount of points. For D-level functionality, you may do this in any tool.

2.1.2 C Level Functionality

Complete the D level functionality using D3 version 5.

2.1.3 B Level Functionality

Complete the C level functionality and implement details-on-demand interactivity.

For example, show details when the mouse hovers over a region and/or symbol. You should implement this in base D3—please do not use a tooltip library. Your details should be readable and informative. Think about which details will be useful and adjust the formatting of the text as appropriate.

2.1.4 A Level Functionality

Complete the B level functionality and implement one of the following:

  • Zooming AND Panning: Allow users to zoom in on a region (either mouse scroll or click) and pan (drag) the view while zoomed in. This will work with any of the visualization techniques, but it can be tricky to properly scale the size of symbols depending on the approach you take.

  • Brushing OR Filtering: Allow users to either brush or filter the data. Choose a column in the dataset with categorical or discrete data,  and then determine how to control which information is brushed/filtered based on that column. Be creative. For example, you could add a linked visualization/legend to control what data is brushed/filtered, or add HTML form controls. Alternatively, if using a symbol map, you could brush/filter other symbols based on a shared characteristic of the symbol clicked on.

You only need to choose one of the above options, but they should be in addition to details-on-demand interactivity.

2.2 Writeups

The following writeups are required to earn credit, regardless of the level of functionality implemented:

  • Dataset: Include information about the original dataset, including its source, size, license, when you accessed it, and any filtering or processing you performed to visualize that data.

  • Attempted Grade: Include your self-assessment of the letter grade you should earn (D, C, B, or A). Clearly indicate how your submission meets the functionality requirements. You do not need to indicate whether you should earn a plus or minus grade.

  • Author Information: Include your professional information, including your name, photo, USF email address, major, expected graduation date, and brief bio about yourself.

For each visualization (not prototype) included in your submission, you must include the following:

  • Encoding: Include how to interpret your visualization, indicating exactly how data is encoded. For example, "passenger count is encoded by the length of each bar, so longer bars represent higher passenger count."

  • Interactivity: Include how to interact with your visualization. For example, "hover over each bar to see detailed information about that airline." If you are unable to interact with your visualization, still include this section and state as much.

  • Wrangling: Include any per-visualization data wrangling you performed on the original dataset, including filtering and processing, and how you wrangled the data (e.g. Tableau, Excel, using the website interface). Link to the wrangled data in your repository when possible.

  • Conclusion: Include one conclusion you can make about the data using your visualization. The conclusion should ideally be something that a simple statistic could not replicate. For example, "airline X has the highest passenger count over this time period" is a simple conclusion that will meet this requirement, but may not earn a plus-level letter grade.

  • Attribution: Include any inspirations that helped you create this visualization, any additional datasets you used, and any code that you referenced. If you have nothing to list, still include this section and state as much.

The writeups do not need to be long—often 1 paragraph with 3 to 5 sentences will suffice.

2.3 Plus/Minus Grades

It is up to the discretion of the grader to assign a minus letter grade (e.g. B-), normal letter grade (e.g. B), or plus letter grade (e.g. B+). The following will be considered in making this determination:

  • Are the functionality requirements fully implemented and reasonably bug-free?

  • Are all of the relevant details provided with the visualizations (such as legends, titles, annotations)?

  • Are all of the required writeups (such as how to interpret your visualization) included?

  • Are the writeups reasonably well-written (with no misspellings and decent grammar).

  • Are the inspirations, sources, and references properly attributed?

  • Is the website well-designed and easy to navigate?

All of the above must be true to qualify for a plus-level letter grade.
