Homework 4 Requirements

For this homework assignment, you must visualize the data from the Fire Department Calls for Service dataset using hierarchical data visualization techniques in D3 version 5. You must perform some data wrangling for this assignment to transform the data into a hierarchical format.

1. Dataset

You must visualize the SF 311 dataset available at:

https://data.sfgov.org/Public-Safety/Fire-Department-Calls-for-Service/nuek-vuh3 (Links to an external site.)

However, the dataset is not hierarchical. You must decide how to form a hierarchy from this dataset. See below for details.

1.1 Hierarchy

You must form a hierarchy from this dataset such that:

  • The hierarchy must have height 3 or higher (i.e. at least 3 levels, including root and leaves).

  • Each leaf node must have an attribute associated with it.

This hierarchy can be formed using dates, regions, and/or categories (call type group, call types, etc.).

It is possible to create a hierarchy by region only, but it is difficult. You can try Division » Battalion » Station (see breakdown (Links to an external site.)). Or, you can try Supervisor District » Zip Code (Links to an external site.), but some zip codes overlap districts. The dataset does use Analysis Neighborhoods (Links to an external site.), which can be mapped to census tracks (Links to an external site.) as well.

It might be interesting to combine different ways to form the hierarchy. For example, you could use Supervisor District » Category » Subcategory as the hierarchy and calculate per-district per-capita (Links to an external site.) incident rates.

Node attributes can be the count of unique incident numbers associated with the node, average or median response time, per sqft incident rate, and so on.

1.2 Wrangling

You may use any tool or language to wrangle the dataset, including the SF Data interface itself. Specifically, students can use the "Filter" » "Sort & Roll-Up" » "Roll-Ups & Drill-Downs" functionality to start building a hierarchy from the dataset.

Either way, I recommend you focus on getting the data into a table/CSV format and then use d3.stratify (Links to an external site.) to put it into the final hierarchical format required by D3. You need (1) a unique node identifier, (2) a way to determine the unique identifier of the parent node, and (3) some value for each leaf node.

2. Requirements

To earn credit, you must both satisfy one of the functionality levels below and provide the required writeups.

 Letter  Requirement
D Wrangle data into a hierarchical JSON format.
C The D-level requirements plus implement one node/link diagram with details-on-demand interactivity in D3v5.
B The C-level requirements plus implement one space filling diagram with details-on-demand interactivity in D3v5.
A The B level requirements plus filtering, brushing, or zooming for one of the visualizations.

Prototyping is not required.

The homework website must have clear navigation to the required sections. Visualizations should be within 960 by 500 pixels in size, although some exceptions are reasonable. 

2.1 Functionality

The functionality by letter grade is as follows:

2.1.1 D Level Functionality

Successfully wrangle the data into a hierarchical format that can be interpreted by either d3.hierarchy() (Links to an external site.) or d3.stratify() (Links to an external site.). The hierarchy must have height 3 or more (i.e. at least 3 levels including the root and leaf nodes) and each leaf node must have an attribute.

2.1.2 C Level Functionality

Complete the D-level functionality and implement a node link visualization using the d3.cluster() (Links to an external site.) or d3.tree() (Links to an external site.) layout. You may use any Coordinate system (Cartesian or polar) and any edge style (straight or curved).

The node attribute must be encoded somehow (i.e. using color, size, shape, etc.). Nodes must have a minimum details-on-demand interactivity showing the node label.

2.1.3 B Level Functionality

Complete the C-level functionality and implement a space-filling or enclosure visualization using d3.partition() (Links to an external site.), d3.treemap() (Links to an external site.), or d3.pack() layouts. You may use any Coordinate system (Cartesian or polar).

The node attribute must be encoded somehow (i.e. using color, size, shape, etc.). Nodes must have a minimum details-on-demand interactivity showing the node label.

2.1.4 A Level Functionality

Complete the B-level functionality and implement filtering, brushing, or zooming in either of the two hierarchical visualizations you created. You only need to implement this interactivity in one of the visualizations, but must still support details-on-demand interactivity in that visualization.

2.2 Writeups

The following writeups are required to earn credit, regardless of the level of functionality implemented:

  • Dataset: Include information about the original dataset, including its source, size, license, when you accessed it, and any filtering or processing you performed to visualize that data.

  • Attempted Grade: Include your self-assessment of the letter grade you should earn (D, C, B, or A). Clearly indicate how your submission meets the functionality requirements. You do not need to indicate whether you should earn a plus or minus grade.

  • Author Information: Include your professional information, including your name, photo, USF email address, major, expected graduation date, and brief bio about yourself.

For each visualization (not prototype) included in your submission, you must include the following:

  • Encoding: Include how to interpret your visualization, indicating exactly how data is encoded. For example, "passenger count is encoded by the length of each bar, so longer bars represent higher passenger count."

  • Interactivity: Include how to interact with your visualization. For example, "hover over each bar to see detailed information about that airline." If you are unable to interact with your visualization, still include this section and state as much.

  • Wrangling: Include any per-visualization data wrangling you performed on the original dataset, including filtering and processing, and how you wrangled the data (e.g. Tableau, Excel, using the website interface). Link to the wrangled data in your repository when possible.

  • Conclusion: Include one conclusion you can make about the data using your visualization. The conclusion should ideally be something that a simple statistic could not replicate. For example, "airline X has the highest passenger count over this time period" is a simple conclusion that will meet this requirement, but may not earn a plus-level letter grade.

  • Attribution: Include any inspirations that helped you create this visualization, any additional datasets you used, and any code that you referenced. If you have nothing to list, still include this section and state as much.

The writeups do not need to be long—often 1 paragraph with 3 to 5 sentences will suffice.

2.3 Plus/Minus Grades

It is up to the discretion of the grader to assign a minus letter grade (e.g. B-), normal letter grade (e.g. B), or plus letter grade (e.g. B+). The following will be considered in making this determination:

  • Are the functionality requirements fully implemented and reasonably bug-free?

  • Are all of the relevant details provided with the visualizations (such as legends, titles, annotations)?

  • Are all of the required writeups (such as how to interpret your visualization) included?

  • Are the writeups reasonably well-written (with no misspellings and decent grammar).

  • Are the inspirations, sources, and references properly attributed?

  • Is the website well-designed and easy to navigate?

All of the above must be true to qualify for a plus-level letter grade.

No submission for this assignment is required; all created repositories will be graded.

If work is committed after the deadline, the repository will be rolled back to the last commit before the deadline prior to grading.
